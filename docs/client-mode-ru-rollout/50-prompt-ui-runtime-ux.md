# Prompt E. UI-интеграция runtime, локализация и UX

## Для кого

Этот prompt предназначен для отдельного агента GPT-5.4 с максимальным effort. Агент отвечает за то, чтобы новые runtime-возможности и i18n-слой стали реальным пользовательским интерфейсом внутри текущей PyQt6-оболочки.

## Обязательный контекст перед стартом

Прочитай:

1. `docs/client-mode-ru-rollout/00-master-spec.md`
2. `docs/client-mode-ru-rollout/01-parallel-workstreams.md`
3. runtime-контракты потока A
4. i18n-контракты и каталоги потока D
5. help/content-структуру потока F, если она уже существует

Текущая UI-база:

- `app/ui/main_window.py`
- `app/ui/detail_panel.py`
- `app/ui/card_view.py`
- `app/ui/sidebar.py`
- `app/ui/dialogs.py`
- `app/ui/settings.py`
- `main.py`

## Твоя зона владения

Можно свободно менять:

- `app/ui/main_window.py`
- `app/ui/detail_panel.py`
- `app/ui/card_view.py`
- `app/ui/sidebar.py`
- `app/ui/dialogs.py`
- `app/ui/settings.py`

Можно точечно менять:

- `app/models.py`, если требуется убрать UI-строки из моделей

Не трогай без крайней необходимости:

- `app/runtime/adapters/*`
- `app/db.py`
- build scripts
- README/help content source files

## Главная цель

Сделать так, чтобы пользователь увидел новый продуктовый режим не в абстрактной архитектуре, а прямо в интерфейсе:

- реальные действия `Подключить`, `Отключить`, `Сделать основным`
- отдельный runtime-статус на карточках
- блок `Подключение` в detail panel
- лог, история и human summary
- настройки client mode и языка
- переключение `Русский / English`
- встроенную справку

При этом нужно сохранить существующую трёхколоночную архитектуру и тёплую визуальную систему, но сделать формулировки и визуальную иерархию немного строже.

## Жёсткие требования

- reachability и runtime status должны отображаться раздельно
- все пользовательские строки должны идти через `tr(...)`
- не должно остаться новых hardcoded English-строк в UI
- общая трёхколоночная структура не меняется
- интерфейс не должен выглядеть как “новый другой продукт”; нужна эволюция текущего UI

## Что именно нужно реализовать

### 1. Инициализация сервисов

В `main.py` и/или `MainWindow`:

- инициализировать translator/localization service
- инициализировать `RuntimeManager`
- связать shutdown cleanup при закрытии приложения
- при смене языка дать UI возможность обновиться без перезапуска, если архитектура позволяет

Если полный live-refresh слишком дорог, допустим аккуратный controlled refresh через `retranslate_ui()` и пересборку текстов.

### 2. Карточки и список

В `CardView` добавь отдельный runtime статус.

Нужно показать состояния:

- `Не подключено`
- `Запуск…`
- `Подключено`
- `Основное`
- `WireGuard активно`
- `Ошибка`

Reachability остаётся отдельным индикатором/диагностикой и не подменяет этот статус.

### 3. Правая панель: новый блок `Подключение`

В `DetailPanel` добавь новый runtime block выше `TCP Reachability`.

В блоке должны быть:

- движок
- текущее состояние
- кто сейчас основной маршрут
- локальный HTTP-порт
- локальный SOCKS-порт
- время запуска
- последняя активность
- последнее рукопожатие
- последняя ошибка
- краткий живой лог
- человеческое summary над техническим логом

Нужные действия:

- `Подключить`
- `Отключить`
- `Сделать основным`
- `Открыть полный журнал`
- `Скопировать локальный адрес`

Подумай об enable/disable состояниях:

- `Подключить` выключена, если entry unsupported или уже запускается
- `Отключить` выключена, если сессия не активна
- `Сделать основным` выключена, если entry не подключён или уже primary

### 4. Главное меню и toolbar

Добавь действия:

- `Подключить выбранный`
- `Отключить выбранный`
- `Сделать основным`
- `Остановить все подключения`
- `Журналы`
- `Сеансы`

Верхний toolbar должен оставаться компактным и не ломать существующую навигацию.

### 5. Settings

Расширь `SettingsDialog`:

- `Режим клиента`
- `Восстанавливать подключения при запуске`
- `Очищать системный proxy при выходе`
- `Сворачивать в трей`
- `Автопереподключение`
- `Сколько строк хранить в журнале`
- `Папка движков`
- `Язык интерфейса`

Нужно сделать это аккуратно, не превращая настройки в хаотичную форму.

### 6. Language switch

Пользователь должен видеть полноценный переключатель:

- `Русский`
- `English`

Ожидания:

- новый язык применяется последовательно
- не остаётся смешанных половинчатых фрагментов
- критические runtime/status messages тоже переключаются

### 7. Welcome и Help shell

Сделай:

- обновлённый welcome/quick start flow
- окно встроенной справки

Контент должен подтягиваться из i18n/help resources, а не быть жёстко зашитым в код диалога.

### 8. Human-friendly error presentation

В runtime/log UI над raw log должен показываться человекопонятный блок:

- что произошло
- успешно ли подключение
- что можно сделать дальше

Если ошибка техническая, сначала дай нормальную короткую русскую фразу, потом detail.

### 9. Тон и copy

Сделай формулировки:

- короткими
- однозначными
- дружелюбными
- чуть более собранными, чем текущие

Избегай:

- игривых расплывчатых подписей
- канцелярита
- чрезмерно техничных заголовков без пояснения

## Дизайн-ограничения

- не ломать тёплую цветовую систему без необходимости
- сохранять текущую композицию окна
- при добавлении новых статусных плашек держать визуальную иерархию спокойной
- не превращать правую колонку в перегруженный техпанелью столб

## Что не нужно делать здесь

- реализовывать реальные subprocess adapters
- переписывать build scripts
- заново проектировать всю тему приложения

## Тесты и проверки

Минимум нужно покрыть:

- ключевые окна рендерятся без missing keys
- language switch меняет подписи меню/settings/runtime block
- unsupported `OTHER` entry показывает понятное disabled-state объяснение
- runtime status не конфликтует с reachability status

Если UI-тесты сложно автоматизировать полностью, добавь хотя бы smoke checks и структуру для ручного acceptance.

## Acceptance criteria

Работа считается готовой, если:

- весь основной UI переведён на `tr(...)`
- в карточках и detail panel есть отдельный runtime-статус
- есть runtime actions в меню/toolbar/panel
- есть settings для client mode и языка
- welcome/help встроены в приложение
- интерфейс ощущается более собранным, но остаётся “тем же ProxyVault”

## Формат финального ответа агента

В конце сообщи:

1. Какие UI-модули изменены
2. Какие новые surface areas появились
3. Как реализован language switch
4. Какие hardcoded строки были вычищены
5. Какие ограничения ещё зависят от незамёрженных runtime adapters
