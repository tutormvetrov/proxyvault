# Prompt D. i18n-платформа, ключи и каталоги `ru/en`

## Для кого

Этот prompt предназначен для отдельного агента GPT-5.4 с максимальным effort. Агент строит локализационный фундамент, на который потом опирается UI-поток.

## Обязательный контекст перед стартом

Прочитай:

1. `docs/client-mode-ru-rollout/00-master-spec.md`
2. `docs/client-mode-ru-rollout/01-parallel-workstreams.md`

Изучи текущие пользовательские строки как минимум в:

- `app/ui/main_window.py`
- `app/ui/detail_panel.py`
- `app/ui/dialogs.py`
- `app/ui/settings.py`
- `main.py`
- `app/models.py` там, где есть display labels/status labels

## Твоя зона владения

Можно свободно менять:

- `app/i18n/*`
- новые help/catalog resources, если они относятся к локализации

Можно точечно менять:

- `app/models.py`, если нужно убрать display labels из модели в словари

Не трогай без крайней необходимости:

- `app/ui/*`
- runtime adapters
- build scripts

## Главная цель

Создать полноценный i18n-layer, а не “разовый перевод строк”, чтобы UI-поток мог механически перевести весь интерфейс на `tr(...)`.

## Жёсткие требования

- русский язык является default locale
- английский является полноценным альтернативным locale
- локализация должна иметь устойчивую структуру ключей
- отсутствие ключа должно быть заметно в тестах и в debug-сценарии
- новый слой не должен зависеть от конкретных виджетов PyQt

## Что именно нужно реализовать

### 1. Базовые сущности

Нужны:

- `SupportedLocale` enum или эквивалентный registry
- `Translator` / `LocalizationService`
- глобально понятный `tr(key: str, **params) -> str`
- каталоги `ru` и `en`
- централизованный список ключей или key registry

### 2. Структура ключей

Нужно ввести и задокументировать стабильную схему ключей.

Минимальные namespaces:

- `app.*`
- `menu.*`
- `toolbar.*`
- `action.*`
- `common.*`
- `settings.*`
- `runtime.*`
- `reachability.*`
- `dialog.*`
- `onboarding.*`
- `help.*`
- `error.*`
- `toast.*`

Избегай хаотичных ключей вроде `label1`, `misc.error2`, `foo.bar`.

### 3. Полные словари `ru` и `en`

Нужно подготовить полный набор строк хотя бы для:

- меню
- toolbar
- карточек
- detail panel
- settings
- welcome
- import/export flows
- runtime states
- reachability states
- help texts
- human error explanations

Русские тексты должны быть:

- литературными
- короткими
- понятными человеку без техбэкграунда
- дружелюбными, но собранными

Английские тексты должны быть:

- не машинным переводом плохого качества
- смыслово синхронными с русскими

### 4. Технические display labels из моделей

Сейчас часть строк живёт в `app/models.py` как display/status labels.

Нужно решить аккуратно:

- либо убрать пользовательские строки из доменных моделей в i18n
- либо оставить только machine-state в модели, а отображение отдавать через translator

Предпочтительно второе.

### 5. Missing key strategy

Реализуй понятное поведение при отсутствии ключа:

- для тестов missing key должен быть заметен
- для разработки можно возвращать что-то вроде `!!missing:key!!`
- не должно быть молчаливого fallback на пустую строку

### 6. Runtime/localization helpers

Подготовь helpers для форматирования:

- относительного времени
- длительности
- статусов runtime/reachability
- коротких человеческих summary lines

Важно:

- не жёстко прошивать английские слова в `format_relative_time` и аналогичные функции
- дать UI-потоку переиспользуемые форматтеры

## Что не нужно делать здесь

- массовый рефакторинг всех UI-виджетов на `tr(...)`
- runtime manager
- real engine adapters

## Очень желательные артефакты

Если успеваешь, добавь:

- отдельный файл с inventory/mapping текущих hardcoded строк на новые ключи
- компактную документацию по использованию `Translator`

## Тесты, которые нужно добавить

Минимум:

- translator возвращает русский locale по умолчанию
- переключение на English работает
- параметризованные строки подставляют значения корректно
- missing key поведение явно заметно
- каталоги `ru` и `en` содержат одинаковый обязательный набор ключей

## Acceptance criteria

Работа считается готовой, если:

- `app/i18n/` существует
- есть стабильная key taxonomy
- есть полноценные словари `ru` и `en`
- русский выбран по умолчанию
- отсутствующие ключи не теряются бесшумно
- тесты на translator/catalog parity проходят

## Формат финального ответа агента

В финале сообщи:

1. Какие i18n-модули созданы
2. Как устроена key taxonomy
3. Как обрабатываются missing keys
4. Какие форматтеры вынесены из существующих моделей
5. Что должен сделать UI-поток для полной интеграции
